## 一、项目背景

### 1.1 政策依据

根据《"十五五"体育彩票渠道发展实施纲要》（征求意见稿）："积极探索与多业态融合发展。在零售、休闲娱乐、文旅等多业态高频服务场所积极增设'体彩服务角'，提供彩票销售、购彩咨询等服务，依托多业态流量优势扩大体育彩票服务覆盖面，拓宽体育彩票消费场景。"此项促进渠道业态多元化发展的工作已列入国家体育总局体育彩票管理中心2026年重点工作内容，是优化体育彩票渠道布局的关键举措，也是助力体育公益事业可持续发展的重要支撑。

### 1.2 项目定位

福建省体育彩票管理中心拟于2026年实施新场景售彩项目，项目开展包括制定试点方案及试点准备、试点营业及项目总结等全生命周期。项目具备短期落地价值与长期增长潜力，不仅有效拓宽购彩入口、提升购彩便利性，实现体彩销售收入增长，同时优化渠道运营结构带动合作业态客流提升，实现体彩机构与合作商家的互利共赢。

### 1.3 项目约束

- **项目最高限价：** 人民币1,450,000元
- **项目服务期：** 合同生效之日起壹年
- **采购方式：** 公开招标

## 二、项目目标

### 2.1 总体目标

建设体彩与多业态融合的新场景售彩模式，实现从"启动—落地—运营"的高效执行，服务覆盖方案编制、软件开发、门店建设、人员培训、运营监控、复盘总结、标准输出的全生命周期管理。

### 2.2 量化指标

| 目标维度 | 量化指标 | 说明 |
|---------|---------|------|
| 合规率 | 100% | 方案完全符合《彩票管理条例》及监管要求 |
| 可落地性 | 100% | 每个场景具备明确的选址标准、建设流程、人员配置和预算 |
| 可复制性 | ≥90% | 运营规范标准可在同类业态场景中推广 |
| 创新性 | ≥80% | 业态创新、技术方案创新、运营模式创新 |
| 设备正常运行率 | ≥99.9% | 门店终端设备稳定运行 |
| 网络连通率 | 100% | 门店内网与云端通信不中断 |
| 监控存储 | ≥30天 | 门店监控录像存储可回放、清晰可辨 |
| 验收通过率 | 100% | 形象布建验收全部通过 |

## 三、项目范围

### 3.1 业务范围

| 范围项 | 要求 |
|-------|------|
| 业态场景 | ≥2类差异化业态（休闲娱乐、文旅景区），区别于传统店、便利连锁店等常见业态 |
| 门店建设 | ≥10个标准化新场景布建 |
| 软件开发 | 体彩专属即开票营销管理平台（客户端+后台管理系统+收银台终端） |
| 运营监控 | 常态化运营监控+季度专项报告 |
| 人员培训 | 营业前1轮+营业后≥2轮，6类培训课件 |
| 复盘总结 | 运营规范标准编制+项目总结报告 |

### 3.2 排除范围

| 排除项 | 说明 |
|-------|------|
| 其他彩种 | 仅售卖体彩顶呱刮即开票，不涉及乐透型、竞猜型等彩种 |
| 互联网远程售彩 | 禁止接入外网开展直接交易，坚决杜绝互联网远程售彩违规行为 |
| 现有渠道改造 | 不涉及传统体彩店、便利连锁店等现有渠道的改造 |
| 体彩中心系统改造 | 不改造体彩中心现有核心系统，仅通过数据接口对接 |

## 六、系统架构

### 6.1 架构设计原则

| 原则 | 说明 |
|------|------|
| 内网闭环 | 所有客户端均部署在门店内网，不直接暴露到公网，交易指令在门店内网发起 |
| 授权准入 | 客户端必须通过设备证书+账号双重认证后才可连接服务端 |
| 代理安全通道 | 内网通过VPN/mTLS加密隧道代理到公有云，防止中间人攻击 |
| 集中管控 | 服务端统一部署在公有云，便于运维、升级和数据汇总 |
| 合规优先 | 坚决杜绝互联网远程售彩，所有交易在门店物理空间内闭环完成 |
| 高可用 | 关键服务冗余部署，单点故障不影响整体服务 |

### 6.2 系统架构

系统采用三层架构设计：

**门店内网层：** 顾客点单终端（Android平板）、收银台终端（Android/Windows）、直播手机、监控摄像头，所有终端通过内网安全代理网关统一出口。

**安全传输层：** 内网安全代理网关 → VPN加密隧道（WireGuard/IPSec）→ mTLS双向认证，建立门店到云端的加密通信通道。

**公有云服务层：** Nginx负载均衡 → 业务API服务集群 → MySQL数据库（主从复制）+ Redis缓存 + 对象存储OSS，以及管理后台Web和数据报表服务。

### 6.3 逻辑架构

| 层次 | 组件 | 说明 |
|------|------|------|
| 展示层 | 顾客点单终端、收银台终端、管理后台 | 各终端的UI展示和交互 |
| API网关层 | Nginx反向代理、认证鉴权过滤器、限流熔断器 | 请求路由、认证校验、流量控制 |
| 业务服务层 | 用户与权限、商品管理、订单交易、库存管理、直播管理、终端管控、结算对账、数据报表 | 8大核心业务服务 |
| 数据存储层 | MySQL、Redis、RabbitMQ、OSS | 持久化存储与消息中间件 |
| 基础设施层 | VPN/mTLS安全通道、ELK日志采集、Prometheus监控、CI/CD流水线 | 运维支撑体系 |

### 6.4 技术选型

| 层次 | 技术组件 | 选型 | 选型理由 |
|------|---------|------|---------|
| 后端框架 | Spring Boot 3.x | Java生态成熟，团队经验丰富 |

### 7.2 安全通道设计

**VPN隧道加密方案：**

| 项目 | 方案 |
|------|------|
| 隧道协议 | WireGuard（优先）或IPSec |
| 加密算法 | ChaCha20-Poly1305（WireGuard）/ AES-256-GCM（IPSec） |
| 密钥交换 | Curve25519（WireGuard）/ DH Group 14（IPSec） |
| 密钥轮换 | 每24小时自动Rekey |
| 隧道监控 | 心跳检测60秒间隔，3次丢失触发重连 |

**mTLS双向认证方案：**

| 项目 | 方案 |
|------|------|
| 证书格式 | X.509 v3 |
| 密钥算法 | RSA 2048 / ECDSA P-256 |
| 证书有效期 | 1年，到期前30天自动续期 |
| CA管理 | 自建私有CA，离线根CA+在线中间CA |
| CRL更新 | 证书吊销后1小时内更新CRL |
| 证书存储 | 设备端加密存储，私钥不可导出 |

### 7.3 合规边界界定

| 行为 | 是否合规 | 说明 |
|------|---------|------|
| 顾客在门店内通过点单终端自助下单 | 合规 | 交易在门店内网发起 |
| 销售员在门店收银台代客下单 | 合规 | 交易在门店内网发起 |
| 顾客通过手机APP远程下单 | 违规 | 属于互联网远程售彩 |
| 非授权终端通过互联网访问交易接口 | 违规 | 云端不直接开放交易接口 |
| 门店终端通过VPN隧道连接云端处理交易 | 合规 | 交易发起端在门店内网 |
| 顾客在门店外扫码访问点单页面 | 违规 | 点单终端仅在内网可访问 |

**技术保障措施：**

- 云端API不直接暴露公网，仅接受VPN隧道内的请求
- 所有API请求必须携带有效的设备证书和用户Token
- 请求IP必须属于已注册门店的VPN出口IP白名单
- 点单终端和收银终端仅在内网环境运行，无公网访问能力

### 7.4 应用安全

| 安全威胁 | 防护措施 |
|---------|---------|
| SQL注入 | 参数化查询+ORM框架+输入白名单校验 |
| XSS攻击 | 输出编码+Content-Security-Policy头+HttpOnly Cookie |
| CSRF攻击 | CSRF Token+SameSite Cookie+Referer校验 |
| 接口篡改 | 请求签名（HMAC-SHA256）+时间戳防重放 |
| 暴力破解 | 登录失败5次锁定15分钟+验证码 |
| 越权访问 | RBAC权限校验+数据范围过滤 |
| 敏感信息泄露 | 日志脱敏+错误信息通用化+调试模式关闭 |
